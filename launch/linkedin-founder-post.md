# LinkedIn founder post draft

I am publishing the public repository surface for TransformIA Capsule Launcher.

TransformIA Capsule is being built around a practical belief: business AI should become more useful without becoming less governed. That is especially important for SMEs and operators in Spain, Europe and other markets where trust, data boundaries and implementation clarity matter.

The new public repo is not a runtime release. It is a source-available launcher surface that makes the product direction inspectable:

- README and quickstart validation for first visitors.
- Public-safe template manifest catalog.
- Placeholder-only local/BYOK configuration scaffold.
- Documentation for the private core boundary.
- Documentation for future Cloud handoff boundaries.
- Validators that block false launch claims.

The private TransformIA runtime remains private. The public repo does not include customer data, connector workflows, proprietary orchestration logic, executable templates, payment flows, Cloud sync or telemetry.

The local/BYOK path is planned and documented because many teams want to understand control, key ownership and deployment posture before they trust AI workflow systems. The config scaffold is intentionally placeholder-only today.

I am looking for feedback from builders, operators, SMEs and governance-minded teams:

- Is the boundary between public launcher and private runtime clear?
- Are the template manifest categories useful?
- What would you need before adopting a future governed launcher?
- Which claims should remain blocked until implementation is real?
