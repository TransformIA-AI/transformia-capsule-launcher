# README Badges and Discovery Posture

## Purpose

Badges help first visitors scan the repo quickly, but they must remain static and truthful.

## Allowed badge posture

- Static status badge for `v0.6-C02`.
- Static source-available badge.
- Static private-core-not-included badge.
- Static installer-promise-only badge.
- Static local/BYOK planned-path badge.

Plain-text badges are preferred in C02 because they avoid external image dependencies and avoid implying CI, releases or downloads that do not exist.

## Blocked badge posture

- No fake CI badge.
- No fake build passing badge.
- No fake download count.
- No fake version release.
- No production-ready claim.
- No open-source badge.
- No badge implying installer availability before an installer exists.
- No badge implying binary availability before a binary exists.
- No badge implying Cloud, payment, Stripe, subscription, customer portal, auth, DB, telemetry or connector execution exists in this repo.

## Discovery posture

README discovery wording should emphasize:

- public distribution surface;
- local/BYOK/control promise;
- source-available/IP boundary;
- private core not included;
- quick validation path;
- future TransformIA Cloud handoff;
- next Atlas step.
